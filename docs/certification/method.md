# Certification method

Certification has four non-interchangeable layers:

1. repository conformance for pure logic, parsers, packages, examples, and
   deterministic browser behavior;
2. headed named-profile runtime scheduling for exact browser-side eligible
   deadlines and canvas submissions;
3. optional observed-display continuity from a qualifying independent trace or
   calibrated high-speed capture; and
4. authored visual-boundary analysis.

A runtime run records exact candidate, fixture, harness, browser, OS, hardware
class, display mode, power state, capabilities, scenarios, raw ledgers, and
ownership counters. It requires three fresh-browser repetitions, 1,000 loop,
route, inverse, and portal boundaries, 10,000 rapid pure inputs plus a 1,000
headed subset, at least 300 throughput outputs, zero format underflow, exact
content identity, normative boundary gaps, and terminal settlement.

The statuses are closed: `passed`, `failed`, `unsupported`, `inconclusive`,
`not-run`, and `withdrawn`. Failed and inconclusive attempts are retained.
Every profile must prove the fatal error boundary: one terminal playback error,
one matching rejected preparation, stable public failure identity, `error`
readiness, and settled runtime ownership. A dedicated canonical JSON ledger is
bound to the exact candidate, fixture, and stable-read `certification.html`
digests, as well as the runtime report ID, canonical public-environment digest,
and derived profile ID. Report-bundle validation recomputes every binding, so a
ledger cannot be replayed into a different run or profile and unrelated
scenario attachments cannot satisfy this criterion. The harness is admitted
to that authority only after its manifest bytes are securely read and verified
with the `browser-harness` role and `text/html` media type. This criterion is
required even when animation is unsupported; the animation claim itself
remains `unsupported`, never `passed`. The boundary criterion certifies AVAL's
failure signaling only. Alternate UI remains wholly owned by the consumer and
is not certification evidence.

The playground resource-fault profile is the browser-side producer for this
ledger. It installs its error observer before connecting the element, retains
the first and repeated `prepare()` rejections by identity, snapshots source
cleanup before final element disposal, and keeps the raw error observer active
through retirement plus a bounded two-task drain. Delayed duplicate fatal
events therefore invalidate the witness. The producer exports strict ledger
input only when every boundary assertion passes. It does not assemble or
promote a runtime report; the external report process canonicalizes,
digest-binds, and attaches that input to the exact candidate report.

Observed-display evidence is separate. External capture must be at least four
times independently measured refresh with calibration and drop detection.
Ambiguity is `inconclusive`. Browser callback and readback timestamps are never
renamed as display or scan-out evidence.
